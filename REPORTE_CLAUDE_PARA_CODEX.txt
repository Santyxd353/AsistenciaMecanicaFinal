2) BUGS CORREGIDOS EN EL FLUJO OFFLINE (WEB / PWA)
============================================================

Contexto: el modulo "offline + sincronizacion" del examen 2 ya estaba en la web
(cola IndexedDB en offline-queue.service.ts + dedup en backend por cliente_sync_id
en solicitudes.py:868-910). Pero tenia 2 fallas reales.

--- BUG 1: la cola no sincronizaba al abrir la app si ya habia conexion ---
Sintoma: si el usuario crea una emergencia offline, cierra la app, y la vuelve a
abrir YA con internet, el evento 'online' del navegador NO se dispara (porque nunca
estuvo offline en esa sesion), asi que las emergencias pendientes quedaban sin
sincronizar para siempre.
Fix (offline-queue.service.ts, constructor): tras refrescar la lista, si
navigator.onLine es true, se dispara un flush() inicial.

    if (navigator.onLine) {
      this.flush().subscribe();
    }

--- BUG 2: la UI del cliente no se actualizaba tras sincronizar ---
Sintoma: al crear offline, client-portal mete un "report falso" con id negativo
(-Date.now()) y estado 'pendiente_sync'. Cuando vuelve la conexion y la cola
sincroniza con el backend (POST real), la UI seguia mostrando ese report falso
(estado pendiente_sync, id negativo) y el realtime no se enganchaba al id real
del backend. Incumplia el requisito "la app actualiza el estado local".
Fix en 2 partes:
  1) offline-queue.service.ts: se agrego un Subject `flushed$` que emite el
     resultado { synced, pending, failed } cada vez que termina un flush
     (automatico al volver online, o manual).
  2) client-portal.component.ts (ngOnInit): se suscribe a offlineQueue.flushed$;
     si synced > 0, muestra mensaje y llama loadReports() -> recarga desde el
     backend, reemplaza los reports falsos por los reales y reengancha el realtime
     al id correcto. La suscripcion se limpia en ngOnDestroy (flushedSub).

Como probarlo:
  - Abrir /cliente como cliente. DevTools -> Network -> Offline.
  - Crear solicitud -> aparece como "pendiente de sincronizacion".
  - Volver a Online -> se sincroniza sola y la lista se refresca al estado real.

3) FEATURE NUEVA: UI DE COTIZACIONES PARA EL CLIENTE (WEB)
============================================================

Contexto: el backend YA tenia el flujo completo de cotizaciones
(backend/app/api/cotizaciones.py) con endpoints accesibles por el cliente:
  - GET  /api/v1/cotizaciones/solicitudes/{id}   -> listar cotizaciones (dueño)
  - POST /api/v1/cotizaciones/{id}/seleccionar   -> cliente elige taller
Y el servicio frontend CotizacionService (core/cotizacion.service.ts) ya existia.
PERO el cliente (client-portal) NO usaba nada de eso -> no podia ver cotizaciones,
ni elegir taller, ni ver el tiempo de reparacion. Eran extras OBLIGATORIOS del
enunciado ("Obtener cotizaciones", "Seleccionar un taller", "Obtener el tiempo
que tardara en repararse").

Que se agrego en client-portal.component.ts:
  - Inyeccion de CotizacionService.
  - Estado nuevo: cotizacionesPorReporte (Record<reportId, Cotizacion[]>),
    cotizacionesLoading, cotizacionMessage, cotizacionError, selectingCotizacionId.
  - Getter pendingReports: reports en estado 'pendiente' o 'buscando_taller'.
  - reloadCotizaciones(): carga cotizaciones de todos los pendientes; limpia cache
    de los que ya no estan pendientes. Se llama al final de loadReports().
  - loadCotizaciones(reportId): GET cotizaciones de una solicitud (filtra estado
    'enviada').
  - seleccionarCotizacion(cotizacion, reportId): POST seleccionar -> el backend
    asigna el taller (solicitud pasa a ASIGNADA), luego loadReports().
  - Manejo de eventos realtime nuevos en applyRealtimeEvent():
      * 'cotizacion.nueva'    -> recarga cotizaciones de esa solicitud (en vivo).
      * 'cotizacion.aceptada' -> recarga reports (reflejar ASIGNADA).
  - Seccion nueva en el template (panel-cotizaciones): por cada solicitud pendiente
    lista las cotizaciones con taller, costo (Bs), tiempo de reparacion (h),
    ETA (min), garantia (dias), repuestos, calificacion; boton "Seleccionar este
    taller".
  - CSS minimo agregado (.cotizacion-list, .cotizacion-card) dentro del styles[]
    inline del componente. Se mantuvo bajo el budget de 10kB del componente.

Como probarlo (flujo completo):
  1. Como cliente: crear una solicitud (queda 'pendiente' / 'buscando_taller').
  2. Como taller (otro login, mismo tenant): emitir una cotizacion para esa
     solicitud (POST /api/v1/cotizaciones/solicitudes/{id}).
  3. Volver al cliente: aparece la seccion "Elige el taller para tu servicio" con
     la cotizacion (en vivo via WebSocket). Click "Seleccionar este taller" ->
     la solicitud pasa a ASIGNADA al taller dueño de esa cotizacion.
