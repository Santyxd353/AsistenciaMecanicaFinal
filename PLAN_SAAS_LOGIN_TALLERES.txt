================================================================================
1. OBJETIVO FUNCIONAL
================================================================================

Convertir el flujo actual de registro/login en un flujo SaaS real por taller/tenant.

Flujo esperado:

1. Una persona entra a la pagina publica.
2. Ve planes disponibles con beneficios.
3. Selecciona un plan.
4. Simula pago del plan.
5. Pasa a crear su taller con el formulario existente.
6. Al crear el taller, queda como administrador de ese taller.
7. Desde el panel del taller puede crear mecanicos y usar el sistema normal.
8. El login se separa en:
   - Login Trabajadores:
     - Buscar taller.
     - Seleccionar taller.
     - Ingresar usuario y contrasena.
     - Entrar como mecanico/trabajador del taller seleccionado.
   - Login Administrador:
     - Solo usuario y contrasena.
     - No pide taller.
     - Entra como administrador de taller o superadmin segun su cuenta.
9. Los planes limitan cantidad de administradores y mecanicos por taller.
10. Si un taller con plan gratis intenta crear mas administradores, se redirige a una pagina de upgrade.

3. DECISIONES DE ARQUITECTURA
================================================================================

3.1 Tenant y taller

Mantener la idea:
- Tenant = organizacion SaaS.
- Taller = unidad operativa dentro del tenant.

Para este proyecto, como el usuario pide "taller" como unidad SaaS, se usara:
- Un tenant por taller registrado desde la pagina de planes.
- El primer administrador creado sera el propietario del taller.

3.2 Roles

Mantener roles existentes para no romper:
- admin: superadmin global si tenant_id es NULL.
- workshop: administrador de taller si tenant_id tiene valor.
- tecnico: trabajador/mecanico.
- driver: cliente.

No crear nuevos roles salvo que sea estrictamente necesario.

Interpretacion:
- Login Administrador acepta:
  - admin global.
  - workshop/admin de taller.
- Login Trabajadores acepta:
  - tecnico.
  - opcionalmente otros roles operativos futuros.

3.3 Planes

Agregar planes SaaS como modelo y logica centralizada.

Planes:

PLAN GRATIS
- 1 administrador por taller.
- 5 mecanicos.
- 30 solicitudes/mes.
- Dashboard basico.
- Tracking en tiempo real.
- Soporte comunitario.

PLAN INTERMEDIO
- 3 administradores por taller.
- 10 mecanicos.
- 200 solicitudes/mes.
- KPIs operativos.
- Cotizaciones.
- Historial avanzado.
- Soporte por correo.

PLAN PREMIUM
- 10 administradores por taller.
- 20 mecanicos.
- Solicitudes mensuales altas.
- Dashboard avanzado.
- Auditoria.
- Reportes exportables.
- SLA configurable.
- Soporte prioritario.

PLAN PRO
- Administradores ilimitados.
- Mecanicos ilimitados.
- Solicitudes ilimitadas.
- Multi-sucursal preparado.
- Analitica avanzada.
- Auditoria completa.
- Integraciones futuras.
- Soporte premium.

3.4 Pago

No usar pasarela real.
Implementar pago mock:
- Estado pendiente.
- Estado pagado.
- Referencia simulada.
- Fecha de pago.

3.5 Login trabajadores con taller

Necesario porque en SaaS puede existir mismo username en distintos talleres.

Implementacion recomendada:
- Mantener User.username unico global por ahora para compatibilidad.
- Agregar campo username_scoped opcional en futuro si se quiere permitir duplicados.
- Para MVP inmediato:
  - Trabajador busca taller.
  - Selecciona taller.
  - Login valida username/password y que user.tenant_id corresponda al tenant del taller seleccionado.

Esto no rompe la BD actual y cumple el flujo.

3.6 Administradores adicionales

Agregar endpoint para crear administradores del taller.
Debe validar:
- Usuario actual debe ser workshop del mismo tenant o admin global.
- Limite del plan.
- Si excede limite:
  - Backend devuelve HTTP 402 o 403 con codigo "PLAN_LIMIT_ADMIN".
  - Frontend redirige a /upgrade-plan.

20. CRITERIOS DE ACEPTACION
================================================================================

Se considera terminado cuando:

1. Existe pagina publica de planes.
2. Cada plan muestra limites y beneficios.
3. Se puede seleccionar plan y hacer pago mock.
4. Despues del pago se puede crear taller.
5. El usuario creador queda como administrador del taller.
6. Login Administrador existe y no pide taller.
7. Login Trabajadores existe y pide buscar/seleccionar taller.
8. Trabajador solo puede entrar si pertenece al taller seleccionado.
9. Plan gratis limita a:
   - 1 administrador.
   - 5 mecanicos.
10. Plan intermedio limita a:
   - 3 administradores.
   - 10 mecanicos.
11. Plan premium limita a:
   - 10 administradores.
   - 20 mecanicos.
12. Plan pro no limita administradores ni mecanicos.
13. Si se supera limite, se redirige a upgrade plan.
14. Todo lo anterior funciona en Docker en:
   - Frontend: http://127.0.0.1:4200
   - Backend: http://127.0.0.1:8000
